# Lab 14: Certainty, Hedging, and Calibration

**Advanced course, Group I: bridge and instruments.** Evidence levels targeted:
`DECODE + OBS + SELF-REPORT`. Optional `CAUSAL` steering is deliberately left
as an extension, not the default run.

## Core question

Does the model carry an internal answerability or certainty signal that is
separate from the hedging words it emits?

This lab builds a reusable uncertainty instrument for the later social,
multi-turn, belief-revision, and self-report labs. The point is not to crown a
single confidence score as the truth. The point is to put three gauges on the
same items and then study their disagreements:

1. **Internal gauge**: residual-stream projection onto a certainty or
   answerability direction. This is `DECODE` evidence.
2. **Distribution gauge**: A/B/C/D option probabilities, entropy, and margin.
   This is `OBS` behavior from the next-token distribution.
3. **Verbal gauge**: generated confidence words such as `certain`, `likely`,
   `unsure`, or `guess`. This is `SELF-REPORT`.

A strong run is not a run where all three gauges agree. A strong run is one
where the agreement and disagreement cases are measured cleanly enough that
you can say what each gauge is allowed to support.

## Why this lab exists here

Labs 12 and 13 asked whether richer concepts survive controls. Lab 14 turns
that same discipline toward uncertainty, because the downstream labs will ask
questions like:

- Does agreement pressure reduce internal certainty or only increase hedging?
- Does a model capitulate while still carrying the original answer state?
- Does self-reported confidence track an internal uncertainty signal?

Those questions are impossible to interpret unless the certainty instrument
has already survived style, format, family, and entropy controls.

## Dataset

The default dataset is:

```text
data/certainty_calibration_items.csv
```

It is generated by `data/make_certainty_calibration.py` and should be frozen in
the repository data manifest. The lab records the observed SHA-256 hash in
`diagnostics/frozen_data_manifest.json`.

The dataset contains three families:

| Family | What it tests |
|---|---|
| `mcq` | Ordinary multiple-choice items plus MCQ-shaped unanswerable controls. |
| `factual_qa` | Known factual questions plus private, future, hidden-information, or unavailable-answer controls. |
| `freeform_answerability` | Short answerability questions rendered into a fixed A/B/C/D frame. |

Every row should include:

| Column | Meaning |
|---|---|
| `item_id` | Stable unique id. |
| `family` | One of `mcq`, `factual_qa`, `freeform_answerability`. |
| `topic` | Topic bucket used for audits. |
| `answerable` | `1` for answerable, `0` for known-unanswerable. |
| `question` | The question stem. |
| `option_a` through `option_d` | Fixed choices. |
| `answer_key` | Correct option letter. Unanswerable items usually use `D`. |
| `expected_answer` | Human-readable answer or unanswerability label. |
| `answer_format` | Optional format note. |
| `confident_statement` | Content-matched confident wording for the style-control direction. |
| `hedged_statement` | Content-matched hedged wording for the style-control direction. |
| `note` | Optional data note. |

### The D-option warning

Most unanswerable items use `D`, often something like “Cannot be determined.”
That is useful for fixed-choice scoring, but it is also a trap: a probe could
learn the answer frame rather than uncertainty. Lab 14 therefore logs
`answer_key_is_d`, option text lengths, and answer-letter baselines. A strong
certainty claim must survive those audits.

### Tier A fallback

For CPU smoke tests, the revised lab includes a tiny built-in fallback dataset
only when the real CSV is missing and the run is clearly Tier A style. The run
manifest marks this as plumbing-only data. Do not use fallback results for a
ledger claim.

## Method

### 1. Render chat prompts and verify the microscope

Lab 14 is an instruct-model lab, so prompts are rendered through the model's
chat template. The run writes a render audit before doing science:

```text
diagnostics/prompt_render_audit.csv
```

The bench self-checks still matter: hook parity and final-depth lens parity
must pass before you trust the residual streams.

### 2. Read the A/B/C/D distribution

For each item, the lab prompts the model to answer with one letter. It then
computes the next-token probability mass for A, B, C, and D, including common
letter-token variants such as `A` and ` A` when they are single tokens.

Artifacts:

```text
tables/answer_distribution_readout.csv
diagnostics/option_token_audit.csv
```

Key columns include option probabilities, entropy, top margin, chosen letter,
correctness, option text lengths, and whether the key is `D`.

### 3. Fit a certainty direction

The certainty direction is a mass-mean direction:

```text
mean(answerable residuals) - mean(unanswerable residuals)
```

The lab uses a stratified train/eval split and selects the stream depth on the
**train split only**. The selection score is control-adjusted:

```text
train real AUC - max(train shuffled AUC, train random AUC, 0.5)
```

Only after depth selection does the run report eval AUC and selectivity. This
keeps the depth curve from becoming a pretty overfitting machine with knobs.

Artifacts:

```text
tables/probe_report.csv
tables/depth_selection.csv
diagnostics/depth_selection.json
```

### 4. Fit a hedging-style direction

The hedging direction is not trained on answerability. It is trained from
content-matched statement pairs:

```text
confident_statement - hedged_statement
```

The revised lab selects a separate hedging depth. It does **not** reuse the
certainty depth by default, because style may emerge at a different part of
the residual stream.

Artifacts:

```text
state/hedging_direction.pt
state/hedging_direction_metadata.json
```

### 5. Test family-held-out generalization

A certainty direction that only works inside one family may be topic,
template, or difficulty. The lab therefore trains on two families and tests on
the held-out family, with shuffled-label and random-direction controls.

Artifact:

```text
tables/family_heldout_generalization.csv
```

### 6. Generate verbal confidence

The model is separately asked to report confidence using a tiny fixed label
set:

```text
certain, likely, unsure, guess
```

Those labels are self-reports. They are not treated as ground truth, and the
parser writes its own guide and parse-source fields.

Artifacts:

```text
tables/verbal_confidence_reports.csv
tables/verbal_confidence_parse_guide.md
```

### 7. Build the disagreement matrix

The run converts each gauge into low/high bins:

- internal projection rank;
- distributional confidence from entropy and margin;
- verbal confidence score.

Then it counts which combinations occur and how accurate the model is in each
cell.

Artifacts:

```text
tables/confidence_signal_table.csv
tables/confidence_disagreement_matrix.csv
tables/disagreement_examples.csv
plots/confidence_disagreement_matrix.png
plots/confidence_signal_correlations.png
```

The disagreement cases are the jewels. Use them in your writeup. Agreement is
comforting; disagreement teaches.

## First artifact-reading path

Start with `certainty_instrument_card.md`. It tells you whether the run is a
usable certainty instrument, a weak signal, or a confounded result.

Then read `diagnostics/depth_selection.json` and `tables/depth_selection.csv`.
Check that the chosen depth was selected on train evidence, not eval evidence.

Next read `tables/probe_report.csv` and `tables/family_heldout_generalization.csv`.
A useful internal direction should beat shuffled and random controls, and it
should not collapse on the held-out family.

Now inspect `tables/length_and_letter_baselines.csv`. If answer-key letter or
option length predicts answerability too well, the probe may be sniffing the
answer frame, not uncertainty.

Then read `tables/confidence_disagreement_matrix.csv` and
`tables/disagreement_examples.csv`. Pick one disagreement case and trace all
three gauges.

Finally read `operationalization_audit.md` before writing claims. The audit is
the tiny courtroom where your favorite interpretation gets cross-examined.

## How to read the plots

### `certainty_probe_by_layer.png`

The top panel is certainty or answerability decodability by stream depth. The
bottom panel is hedging-style decodability. Each panel includes real,
shuffled-label, and random-direction curves. A real curve that only barely
beats controls is not a sturdy instrument.

### `family_heldout_generalization.png`

Each held-out family asks whether the direction learned from the other
families transfers. A failure here usually means topic, format, or difficulty
is still hiding in the direction.

### `reliability_diagram.png`

This compares reported or inferred confidence to empirical accuracy. A point
above the diagonal is underconfident; a point below the diagonal is
overconfident. Read it by gauge: verbal, distributional, and internal
rank-confidence may tell different stories.

### `confidence_disagreement_matrix.png`

Rows combine internal and distributional low/high bins. Columns split verbal
confidence into low/high. The cell text gives count, accuracy, and answerable
rate. A small but clean disagreement cell can be more interesting than a large
agreement cell.

### `confidence_signal_correlations.png`

The left panel compares internal projection to distributional confidence. The
right panel compares internal projection to verbal confidence. Strong
correlation means the gauges move together; weak correlation means Lab 16, 24,
and 25 have something to investigate.

## Evidence discipline

Use `DECODE` for the internal direction:

> A residual-stream direction predicts answerability above shuffled and random
> controls on this dataset.

Use `OBS` for the next-token option distribution:

> The model's A/B/C/D distribution is sharp or flat on this item.

Use `SELF-REPORT` for generated confidence:

> The model says it is certain, likely, unsure, or guessing.

Do not write “the model knows” unless you define the operational sense. This
lab measures fixed-choice answerability and confidence signals, not a private
mental state glowing in a jar.

## Claim templates

A strong positive claim might look like this:

```text
L14-C1 [DECODE]: On MODEL, a residual-stream answerability direction at stream
depth D predicts held-out answerability with eval AUC A and selectivity S over
shuffled labels; mean family-held-out AUC is H. This licenses a fixed-choice
answerability signal, not a general claim about subjective confidence.
Falsifier: held-out family transfer collapses, or answer-letter and length
baselines explain the separation.
```

A strong self-report claim might look like this:

```text
L14-C2 [SELF-REPORT]: On the same items, verbal confidence has ECE E and
correlates with the internal projection at r=R. The disagreement matrix shows
CELL as a concrete case where self-report and internal projection diverge.
Falsifier: hand audit shows the confidence parser mislabels outputs, or a
held-out family shows confidence tracking hedging style rather than accuracy.
```

A clean negative claim is also valuable:

```text
L14-C1 [DECODE, negative]: This run did not validate a reusable certainty
instrument. The real answerability direction failed to beat shuffled/random
controls by the required gap, and family-held-out AUC collapsed on FAMILY.
This is evidence against using the saved direction in downstream belief or
self-report claims without revalidation.
```

## Writeup questions

1. Does the certainty direction generalize across held-out families?
2. Does the certainty direction look more like answerability, option entropy,
   answer-key letter, or hedging style?
3. Find one item where internal projection and verbal confidence disagree.
   Which signal would you trust for deployment, and what would still make you
   nervous?
4. Does verbal confidence look calibrated in `reliability_diagram.png`?
5. Which family is hardest: MCQ, factual QA, or free-form answerability?
6. How should Lab 24 use this instrument when distinguishing capitulation from
   belief revision?
7. How should Lab 25 use this instrument when testing whether self-reported
   confidence is wired to an internal state?

## Common failure modes

| Symptom | Likely cause | What to inspect |
|---|---|---|
| Certainty AUC is high but family-held-out AUC collapses | Topic or family template confound | `family_heldout_generalization.csv` |
| Certainty and hedging projections are nearly the same | Style leakage | `signal_predictiveness.csv`, `probe_report.csv` |
| Internal projection is just option entropy | Distribution confidence, not a new internal gauge | `confidence_signal_correlations.png` |
| Unanswerable items are trivially separable | D-option or answer-text artifact | `length_and_letter_baselines.csv` |
| Verbal confidence parses well but is uncalibrated | Self-report is fluent but unreliable | `reliability_curve.csv` |
| Tier A results look strange | Tiny fallback data or tiny model | `frozen_data_manifest.json` |

## Extensions

### Optional steering, only after validation

After the instrument passes controls, inject the certainty or uncertainty
direction during generation and measure whether the model changes answer
choice, option entropy, hedging words, or only style. This would be a narrow
`CAUSAL` extension. It should use random, shuffled, and hedging-direction
controls on the same dose-response plot.

### Hand-audit verbal confidence

Sample `verbal_confidence_reports.csv`, hand-label the generated reports, and
compare hand labels to parser labels. The parser is a measurement instrument,
not an oracle.

### Downstream bridge

Use the saved certainty and hedging directions in Labs 16, 24, and 25 only if
this lab's own card says the instrument is usable. A weak Lab 14 direction is
not a cute caveat downstream. It is a load-bearing crack.
